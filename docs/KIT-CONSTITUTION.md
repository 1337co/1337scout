# Kit Constitution

Supreme kit-internal reference. Axioms, red lines, authoring methodology, element system, validation policy, evolution discipline. Runtime adapters derive; constitution wins on conflict. Not auto-loaded; consulted on every kit edit, audit, or addition.

---

## Section 1 — Identity

### Purpose

> **The user does not need to know the kit. Talks to scout, follows recommendations, receives finished work within the verified scope of the matched element.** Each element commits to the 100/100 quality bar within its own scope; "gap-free across the whole workflow" is contingent on verifier invocation at substantive closure (Section 5 — gap-free claim discipline).

### What the kit is, and is not

Behavioral discipline encoded into instruction structures. Primes model against vanilla-LLM failure modes (sycophancy, fabrication, premature-done, scope-bleed, format-mirroring, stage-skipping, pressure-collapse, context drift). Model reasoning intact; failure-mode awareness raised at cognitive moments.

**Not** a tool library (adds no capabilities); **not** a domain template (no frameworks/versions in element bodies); **not** a closed rule table (no "if X then Y" lookup); **not** an output format pack (canonical structure is author-side, not output); **not** an intelligence limiter (channels range, does not restrict); **not** a current-state snapshot — timeless methodology and law, while volatile/current facts (versions, history, results) live in the evidence backlog, not element bodies.

### Authority statement

Conflicts within the kit resolve in favor of this constitution.

Authority hierarchy (higher layer wins on conflict):

1. User direct instruction in conversation
2. User global `~/.claude/CLAUDE.md`
3. Workspace `CLAUDE.md`
4. **This constitution (KIT-CONSTITUTION)** — supreme for kit-internal questions
5. Runtime adapters (kit/CLAUDE.md, kit-authoring, other workspace rules)

Adapters diverging from constitution = drift; fix the adapter.

## Section 2 — Principles

### Axioms (13)

**Axiom 1 — Routing/Recommendation Monopoly.** Only the kit's designated routing element produces recommendations as a deliverable; every other element produces an artifact and stops. **Adapter projection exception:** the runtime adapter (kit/CLAUDE.md) projects the routing element's discipline into conversation default — surface a recommendation, ask a blocking question, or affirm direct execution. The adapter does not hand off, brief, or chain. Structured hand-off remains the routing element's deliverable, invoked explicitly via `/scout`.

**Axiom 2 — User Doesn't Need Kit Knowledge.** User invokes the routing element with a single sentence. Routing element diagnoses, briefs the matched element, hands off via structured text. User copy-pastes the chain forward. Kit-internal structure is not a knowledge requirement.

**Axiom 3 — No Constraint, No Direction.** No kit element restricts the model's intelligence or directs its reasoning; the kit primes failure-mode awareness. Discipline-only — it cannot install design taste, which lives as memory, not runtime.

**Axiom 4 — Routing-Centered Architecture, Modular Scaling.** Kit built around its routing element. Every new element integrates cleanly: self-describing frontmatter, fit-evaluable purpose, structured handoff, no hardcoded references. Element addition test (Section 4): "Does adding this break the routing element?" Answer must be no.

**Axiom 5 — Routing Element Reasons with Full Intelligence.** Every recommendation is generated by the model's full reasoning — not rule-templated. Routing element raises failure pattern awareness, inspects state, diagnoses, routes by applying model intelligence to the specific case. **Quality Bar:** best work for element's domain — scope-scaled — user says "exactly what I wanted" — gap-free, error-free, no post-inspection findings.

**Axiom 6 — Routing Briefs, Element Decides.** Routing element analyzes, briefs, hands off — does not pre-decide the matched element's expert domain. Brief carries objectives + constraints + prior decisions + evidence; expert decisions belong to the matched element.

**Axiom 7 — No Pipeline, Universal Domain Fit.** No fixed element sequence imposed; routing is case-by-case (single element, brief chain, or full chain). The kit applies to any domain (code / design / document / data / concept / planning, or similar) with the same methodology.

**Axiom 8 — Elements Discipline, Don't Mechanically Apply Rules.** An element is **not** a separate program, a rule-walker held in fixed instructions, a step-by-step script, or an output template. An element **is** failure-mode awareness priming + structural anchor for cognitive frame + form discipline + boundary marker for escalation.

**Axiom 9 — User Choice: Routed or Direct.** Two paths equally valid: default (routing element → handoff → user copy-paste → matched element) and direct (user invokes element directly). Routing element is convenience-default, not gate-keeper.

**Axiom 10 — Routing-First, Direct Execution Is Last Resort.** Routing element's priority: (1) matched element exists → handoff; (2) no element fits + trivially light → execute directly; (3) no element fits + substantial → ask the user. Silent execution into uncovered territory is scope invention.

**Axiom 11 — All Elements Context-Aware, No Blind Work.** Every element receives sufficient context to reason: skills see conversation history; agents receive self-contained briefing; rules load on path-trigger; hooks receive event metadata; routing element gathers state via tool calls. Insufficient briefing → element guesses.

**Axiom 12 — Universal Quality Bar.** The Quality Bar in Axiom 5 applies to every element — each in its own deliverable shape.

**Axiom 13 — Net Performance Positive.** Kit must not be slower than vanilla. Per-step minimal latency × re-work avoidance = net positive. Hook latency: PreToolUse <1s, SessionStart <500ms. Lazy-load discipline.

### Red Lines (7)

**Red Line 1 — Form Follows Cognitive Load.** No universal section skeleton. Match the form to the moment: an atomic rule is one line; a judgment call is short prose plus its reason; a complex classifier or verifier is structured and detailed. Markdown `##` headers and XML tags carry structure — not `=== ... ===` delimiters, and not an end-of-body reinforcement bookend. Every retained line earns its place by preventing a demonstrated mistake at a named surface (the inclusion test); a line that prevents no named failure is ritualization and drops. Authoring method: `kit/docs/ON-GRAIN-AUTHORING-METHOD.md`.

**Red Line 2 — Positive Framing, Examples Over Labels.** State the positive target first ("do X"); add a "don't Y" only for a real, repeated failure, paired with a concrete reason and a concrete alternative — a solo "do not X" becomes an attention magnet. One concrete good/bad example beats an abstract failure-label; cap examples at 1-3. Reserve NEVER / IMPORTANT for hard safety, tool-routing, or a hard output requirement. State each rule once; RULES use bullets or prose, not numbered+bold.

**Red Line 3 — Intelligence Intact.** No element body restricts model reasoning, no hardcoded count/dimension/amount cap, no closed enumeration, no tier hierarchy imposing fixed order. Discipline channels range through method, not by capping.

**Red Line 4 — Modular Scaling.** Kit designed to preserve compatibility at scale. Hardcode-free discovery (routing via `Grep ^description:`), self-describing frontmatter, no peer-name references, conditional language. Adding a new element costs only that element's file. (Validated up to ~50; 1000-element claim is architectural pending kit-local stress test.)

**Red Line 5 — Discipline, Not Restriction.** Kit's mode is discipline-only — failure pattern priming, structural anchors, form discipline. When count seems necessary, reframe as the failure pattern it prevents.

**Red Line 6 — Hardcode-Free.** Element bodies do not reference other elements by name. No paths beyond `.claude/**`. No methodology brand names ("rebar", "keystone", "anvil") in element bodies. Cross-element references by role permitted. Infrastructure files (settings.json hook bindings, hook script paths) carry paths legitimately as mechanism, not body-level reference.

**Red Line 7 — Principle Over Enumeration.** Element bodies carry general principles, not exhaustive case enumeration. A principle plus a concrete cross-domain example at the point of ambiguity produces discipline; "If X then Y" lookup tables produce mechanical rule-walking.

### Authoring method — On-Grain

The kit's authoring method is **On-Grain Authoring** (`kit/docs/ON-GRAIN-AUTHORING-METHOD.md` — the living doc, consulted before any element edit). It replaces the earlier REBAR method (fixed nine sections + nine reinforcements + `===` delimiters + end-bookends), which over-installed trained dispositions (off-distribution per the instruction-style dig; suppressive per typicality — not the Phase-0 pilot, which is not a valid comparison, see Evidence Backlog).

Principle: write the smallest instruction that prevents a *demonstrated* mistake at the surface where it happens — default behavior is assumed competent in short clean contexts, so the kit anchors at measured decay surfaces and high-blast-radius safety boundaries, it does not re-teach. Inclusion test: every line names the mistake it prevents, or it is cut (safety exception: rare-but-high-cost failures may be anchored preventively). Form, framing, and evidence-routing are governed by Red Lines 1-2 and the method doc; sizing by the Section 3 cap table; rule additions by the Evidence Backlog (Section 4) — these replace the fixed canonical-section and reinforcement-count rules.

**Protected Safety Invariants** (sharpen-only, never weakened or cut on a lean rewrite): credentials handling; destructive/irreversible-operation gating; intent-evidence gate; blast-radius gate; scope-escalation gate; verifier PASS/FAIL + observable-proof discipline; user-authority hierarchy; backup-before-irreversible.

### Negative-framing discipline

A solo "do not X" enumeration becomes an attention magnet (substitution defense empirically required), so On-Grain leads with the positive target and reaches for a prohibition only on a real repeated failure. When a prohibition is genuinely needed, the strongest forms are stative-descriptive ("a verdict re-verified is review") or a failure paired with its named counter in the same statement; a bare imperative is acceptable only when the replacement is obvious and the severity is low. High-severity boundaries are mechanically enforced (stop-gate or hook), not left to a bare imperative.

### Pressure-state axiom (three-push rule)

Second push = user investment. Third push = stronger evidence the position is correct, not weaker.

**Singular placement:** workspace and kit `CLAUDE.md` + verifier agents + the designated routing element. NOT in domain skills (over-triggering against legitimate user iteration), and NOT in a non-verifier / non-routing agent. NOT in adherence hook injection (the always-loaded layer already carries the discipline; duplication leaks into per-turn signal noise).

### Sample-fill discipline

Identity content gets plausible-real fill in voice; BLOCK only on broken-function markers (visible `TODO` / `[bracket]` / "replace this" / `lorem ipsum` / placeholder boxes that break render or execution). Functional sample-fill is expected; broken-function markers are work-shifting disguised as draft.

### Authoring anti-patterns

Beyond Red Lines 1-2: no ALL-CAPS outside a hard-stop, no format-dictation inside the body ("output XML"), no motivational-filler identity, no per-domain template body.

## Section 3 — Element System

### Element catalog (current standard set)

| Element | Role | Designation | Target (On-Grain load-frequency) |
|---|---|---|---|
| `/scout` | Routing element | Routing | 500-1500 (user-invoked skill) |
| `/forge` | Concept developer | Domain | 500-1500 |
| `/chart` | Planner | Domain | 500-1500 |
| `/build` | Executor | Domain | 500-1500 |
| `@tester` | Verifier | Verifier | 1200-3000 |
| `@kit-element-auditor` | Kit-body auditor | Verifier | 1200-3000 (lazy-loaded) |

Kit can grow beyond this baseline, and has — the live set is **12 skills + 10 agents + 7 hooks + 1 MCP** (v5; the table above is the founding baseline, not the live count). New element must pass Element Addition Test (Section 4). Targets are load-frequency tiers, not per-element quotas — a genuinely complex element may exceed its tier when the Evidence Backlog records why.

### Designation rules

**Routing element** — two converging signals: frontmatter `description` names routing/diagnosis AND the element's opening role-claim paragraph states the router/diagnostician role. Only one per kit. When designation holds, the "no unsolicited next-step" prohibition does not apply to that one element. (On-Grain retired the `=== IDENTITY ===` header; the opening paragraph carries the role claim now.)

**Verifier element** — two converging signals: frontmatter `description` names verification/adversarial probing AND the opening role-claim paragraph states the adversarial-verifier role. Typically an agent (isolated context) with `disallowedTools` preventing mutation of verified work. Verdict precedence over creator self-report.

### Token budget governance (load-frequency targets)

Sizing is by **load frequency × complexity**, not a fixed per-element quota. The more often an element loads into context, the harder the pressure to lean it.

| Element class | Target | Notes |
|---|---|---|
| Always-loaded kernel (`kit/CLAUDE.md`) | 1500-2500 tok | hard pressure — loads every turn |
| Auto-loaded rule | 100-400 tok | path-filtered or always-on |
| User-invoked skill (incl. routing) | 500-1500 tok | lazy — loads on invocation |
| Tool-workflow skill | up to 2500 tok | when multi-step tool orchestration justifies it |
| Verifier agent | 1200-3000 tok | lazy — isolated context |
| Complex verifier harness | 3000+ | ONLY if lazy-loaded; record the justification |
| Hook PROSE injection (UserPromptSubmit / SessionStart) | near-zero | the always-loaded layer already carries the discipline |
| Mechanical hook CODE | no prose cap | latency cap instead (Axiom 13) |
| Output style | ~200-400 tok | |
| settings.json | 20000 chars | as needed |
| Description (frontmatter) | 250 chars | 150-220; ≤250 hard when `disable-model-invocation` |

Verify per edit: `wc -m <file>` ÷ 4 (rough chars-to-tokens). Over target without recorded evidence = not finished; but do not force-cut a genuinely complex element that records its justification in the Evidence Backlog (Section 4). This constitution ≤ 7000 tok (governed above the kit layer).

### Five budget categories

Startup auto-load · skill-listing (catalog descriptions ≤250 chars) · invoked-skill body · per-turn hook injection · compaction-survival restore. Each compounds against context budget; cap discipline applies per category, not just per file.

### Hardcode-free discipline

Per Red Line 6: element bodies reference peers by role ("the verifier agent", "the planning skill", "the designated routing element"), not by name; discovery is `Grep ^description:` pattern-matching; conditional language is permitted ("if a designated routing element exists"). Infrastructure files (settings.json hook bindings, hook script paths) carry paths as mechanism, not body reference — not violations.

### Scout-default-mode pattern

The kit-loaded session's default conversation behavior is **silent diagnose-then-recommend**: before non-trivial work, classify internally (is vanilla enough? would an element materially improve it? is ambiguity high enough to ask first?) and make the diagnosis visible only when it changes the next move — recommend an element, ask a blocking question, name a risk, or affirm direct execution. Trivial requests execute directly. Skills/agents are not auto-invoked — the user invokes them; `/scout` is the escape hatch for fresh deep diagnosis when the default has drifted. A default stance, not an enforcement guarantee; reliability reported by kit-local eval.

### 1000-element scaling

Scout discovery (`Grep ^description:`) is linear in count; ≥100 elements wants a top-N relevance filter before full read, ≥1000 a dedicated discovery layer. Architectural claim until kit-local stress test confirms.

### Deployment invariant

Kit runtime is active only when the session opens at a directory whose `.claude/` holds the kit's hooks/rules/skills. **Canonical launch:** open at the kit root. If the session opens at a parent directory above the kit, that directory's `.claude/` is active and the kit's is dormant. Verify the active `.claude/` matches the kit tree before relying on any enforcement claim (hooks, deny patterns, path-filtered rules).

## Section 4 — Authoring Operations

### Element Addition Test (10-point gate)

Before adding any element to the kit, all ten must pass; any failure blocks entry until repaired.

(1) Hardcode-free — no element name refs, no paths beyond `.claude/**`, no methodology brand names. (2) Self-describing description ≤ 250 chars signaling routing fit + skip boundary. (3) Methodology body teaches how to scan/execute/verify, not what to build; no domain spec. (4) On-Grain form — mode-matched to cognitive load, no `===` scaffolding, no end-bookend; every line passes the inclusion test (names the demonstrated mistake it prevents, or is a Protected Safety Invariant). (5) Within its load-frequency target; over-target only with Evidence-Backlog justification. (6) Positive framing first, examples over abstract labels, prohibitions paired with reason + alternative. (7) No pressure-state spread — domain skills and non-verifier/non-routing agents omit the axiom. (8) Designation conditional — both signals (frontmatter description + opening role-claim) converge if claimed. (9) Routing element behavior intact — adding does not break existing routing. (10) Quality Bar measurable — deliverable matches Axiom 12 universal bar.

### Edit operations

Per edit: read before edit; token cap verify (`wc -m / 4`); hardcode scan (peer-name refs, framework/brand names); designation scan if the frontmatter description or opening role-claim changed. Full checklist: the kit-authoring rule.

### Adapter sync rule

Constitution changes may require adapter sync (kit/CLAUDE.md, kit-authoring, other rules). Adapters DERIVE; do not invent governance; shared content (caps, hierarchy, pressure-state placement) lives in ONE place; constitution wins on conflict. `kit/scripts/adapter-sync-lint.sh` catches structural drift but not absence/semantic drift — periodic deep-audit required. Per amendment: "adapter sync needed?" yes → same commit, no → log.

### Drift Decision Tree + three-rule threshold

A newly observed failure: (1) existing principle covers → no edit; (2) sub-form of existing → existing counter sufficient; (3) clusters on a new axis after 3+ recurrences → a new anchored line; (4) last resort → an Evidence-Backlog entry with its surface. **Three-rule threshold:** a failure earns structural priming only after reproducing across three independent tests/sessions/domains. Single observations → notes or local fixes, not constitutional law. **Safety exception:** credential/security/destructive-action failures can earn a stop-gate or hook on a single observation (asymmetric cost).

### Rules without paths frontmatter

A rule without `paths:` is always-loaded. Intentional only when discipline applies regardless of file context. Always-loaded rules document rationale in YAML comment at file-top. Otherwise every rule declares a `paths:` filter.

**Conditional adapter class (runtime-mode rule):** `autonomous-mode.md` — which absorbed the background-output conventions — applies in autonomous/background mode, undetectable by a path-filter, so it stays always-loaded (~580 tok accepted; was a ~889 tok pair pre-merge). Verified debt: Claude Code exposes no `/loop`/scheduled/Task-dispatch discriminator at hook fire time; re-eval if Anthropic later exposes one. Detail: the evidence backlog.

### Deprecation policy

Element no longer earning its place: record its deprecation in the evidence backlog (`role: deprecated`); body removed from `.claude/`; version history updated with deprecation + supersession reference.

**Evolution** (element revised; supersedes prior; version history carries continuity) vs **complete rewrite** (element family changes radically; fresh baseline; old preserved as historical reference).

### Hook authoring discipline

Hooks are shell scripts; On-Grain prose form (mode-match, framing, examples) does not apply to script bodies — they are mechanical code with a latency cap, not prose injection. **Three hook classes:**

- Safety/boundary (PreToolUse on dangerous patterns) → **fail-closed** on cannot-determine-safety, malformed JSON, missing parser, missing helper. Exit 2 + stderr.
- Adherence (SessionStart + UserPromptSubmit, context injection) → **fail-open** with annotated compensation. Exit 0 + stderr note on degraded mode.
- State (PreCompact backup + SessionStart-compact recovery) → **debug-visible** on failure. Exit 1 + stderr; do not block; silent failure breaks pipeline invisibly.

Cross-platform bash 3.2+, single responsibility, bounded output, JSON via Python parser (not sed), idempotent. settings.json hook command paths anchor to `"$CLAUDE_PROJECT_DIR"` — relative paths break on cwd descent.

### Permissions (settings.json)

Two layers: **deny** (bypass-proof) and **ask** (forces prompts in auto). Evaluation: deny → ask → allow. Settings deny is one layer; shell-mediated bypass (`Bash(cat .env)`) requires boundary-guard content scanning as defense-in-depth. Three layers (settings deny + boundary-guard + secret-scanner) — none alone sufficient.

## Section 5 — Validation Policy

### Evidence tiers + validation per edit

**Evidence tier per claim:** proven (live+repeated tests) / partial (mechanism + reasoning audit) / shaky (architecture correct, live evidence weak) / untested (claim only). The evidence backlog's version history annotates element tier.

**Risk-tiered validation:** structural lint (every edit) → self-audit (Element Addition Test, every new element) → isolated agent test (risk-significant element) → multi-domain test (domain skills, 3+ domains) → verifier probe (substantive closures) → routing regression (every scout edit + new description).

**Finished definition:** token cap verified; hardcode scan clean; designation coherent if claimed; routing intact (scout finds the element after rename); adapter sync considered; evidence-backlog record decision made (logged now or marked DEFERRED); risk-appropriate empirical verification present or listed in the untested backlog.

### Reliability claim discipline

No implicit reliability percentages (e.g., "60-70% auto-routing"); operational targets come from kit-local eval runs, recorded in the evidence backlog as evidence-tier classifications. Implicit auto-routing is best-effort; the routing guarantee starts only at explicit `/scout` invocation — scout-default-mode is a stance, not an enforcement.

### Gap-free claim discipline

A "gap-free" / "100% finished" claim requires either verifier invocation at substantive closure OR concrete verification evidence attached (test output, exit code, artifact diff). Self-claim without one is a policy violation; verifier verdict takes precedence over creator self-report (the executor's gap-free claim is contingent on verifier invocation at closure).

### Validation telemetry gaps (untested/shaky tier)

Not claimed as proven until measured: full-chain test on current baseline · 1000-element discovery stress · hook latency p50/p95/p99 · multi-artifact post-compact recovery · tester actually-invoked rate · settings deny FP/FN · 60-turn long-session decay · router over/under-trigger (100-prompt set) · security corpus (200 dangerous/benign). Surfaced as known backlog, not capability.

## Section 6 — Evolution + Records

### Amendment discipline

**Log-worthy** (record the change with its rationale): axiom/red-line add/remove/rewrite; form-discipline structural change; token cap change; authority-hierarchy change; designation-model change; validation-policy change; hook-class-taxonomy change. **Not log-worthy** (a diff suffices): wording, examples, typos, cross-refs, adapter-sync wording. A new axiom or red line requires recorded evidence — an ablation result or a source record — before adoption. The constitution carries current law, not a changelog.
